# Garage MinIO Replacement Evaluation

Last updated: 2026-04-29

## Purpose

This document implements `EGA-35` by defining the first Garage evaluation lane
for Kublai's MinIO exit plan.

Garage is the preferred maintained self-hosted candidate to evaluate before
creating the Boorchu object-store project. Kublai still treats object storage
as an S3-compatible boundary; this evaluation validates whether Garage can
replace MinIO in local, kind, Helm, and self-hosted reference paths without
changing Kublai's application storage contract.

## Current Decision

Status: Garage is accepted for Kublai validation and self-hosted reference
deployments.

Decision rules:

- Use managed S3-compatible object storage for `kublai.com` production.
- Keep MinIO only as temporary legacy validation infrastructure while Garage
  adoption is rolled through CI and deployment docs.
- Do not create the Boorchu repository unless
  `docs/84-garage-operations-licensing-and-migration-decision.md` is
  superseded by a Garage rejection or launch-critical exception.

## Local Validation Path

The first runnable compatibility lane is intentionally separate from the
default Docker Compose stack so MinIO remains stable while Garage is evaluated.

Commands:

```sh
make garage-compatibility-validate
```

The target starts `docker-compose.garage.yml`, bootstraps a single-node Garage
layout, creates a bucket, creates a Garage access key, grants bucket
permissions, and runs the Kublai object-storage integration test against
Garage.

Manual environment after bootstrap:

```sh
ObjectStorage__Endpoint=http://127.0.0.1:3900
ObjectStorage__Bucket=kublai-garage-compat
```

The access key and secret are generated by Garage during validation and are not
committed.

## CI Validation Path

`.github/workflows/garage-compatibility.yml` runs the same
`make garage-compatibility-validate` target on Linux runners when
object-storage code, Garage compose/config, the Garage validator, or EGA-35
evidence changes.

The workflow publishes `docs/reports/garage-compatibility-latest.md` to the
GitHub step summary and uploads it as the `garage-compatibility-report`
artifact. This keeps Garage compatibility evidence separate from the default
MinIO integration lane while the default local fixture is phased out.

## kind And Helm Dependency Selection

The kind HA and Helm certification scripts keep MinIO as the default
validation dependency but can now select Garage or an externally managed
S3-compatible endpoint.

Commands:

```sh
KIND_OBJECT_STORAGE_PROVIDER=garage make kind-ha-validate
HELM_CERT_OBJECT_STORAGE_PROVIDER=garage make helm-certify
```

Provider modes:

| Mode | Purpose | Required Environment |
|---|---|---|
| `minio` | Default temporary local fixture | none |
| `garage` | In-cluster Garage validation dependency | none |
| `external` | Managed S3-compatible provider validation | `*_OBJECT_STORAGE_ENDPOINT`, `*_OBJECT_STORAGE_ACCESS_KEY`, `*_OBJECT_STORAGE_SECRET_KEY` |

The scripts generate runtime Helm overrides for the selected endpoint,
bucket, access key, and secret. `deploy/helm/kublai/values-kind-garage.yaml`
documents the Garage chart override shape; the generated credentials from
Garage bootstrap remain uncommitted.

## Deployment Shape Under Evaluation

Local-only files:

- `docker-compose.garage.yml`
- `deploy/garage/garage.toml`
- `scripts/garage-compatibility-validate.sh`

The included `garage.toml` is single-node, SQLite-backed, and explicitly not a
production topology. Production evaluation must use persistent disks,
documented backup/restore, TLS through ingress or reverse proxy, secret
management, and a multi-node layout if self-hosted durability is required.

## Final Decision Record

`docs/84-garage-operations-licensing-and-migration-decision.md` is the final
EGA-35 decision record. It accepts Garage for Kublai validation and self-hosted
reference deployments, keeps managed S3-compatible storage as the
`kublai.com` production default, and defers Boorchu unless Garage is explicitly
rejected or excepted later.

## Go/No-Go Criteria

Garage has passed the EGA-35 go/no-go gate for Kublai validation and
self-hosted reference deployments:

- the Garage compatibility target exists for CI
- kind and Helm dependency paths have a Garage option
- production preflight has passed against Garage-backed kind/Helm validation
- backup/restore steps for Garage metadata and object data are documented
- AGPLv3 distribution obligations are documented with a conservative release
  posture

Boorchu remains deferred unless Garage is rejected by a future decision record
or launch-critical exception.

## Execution Tranches

`EGA-35` remains the umbrella decision ticket. The execution work is split into
paired implementation and validation tickets:

| Ticket | Purpose | Dependency |
|---|---|---|
| EGA-36 | Expand provider contract tests for Garage, MinIO, and managed S3-compatible stores | EGA-35 |
| EGA-36T | Validate the expanded provider contract test coverage and report content | EGA-36 |
| EGA-37 | Add a Garage-backed GitHub Actions compatibility lane | EGA-36T |
| EGA-37T | Validate Garage CI evidence and report publication | EGA-37 |
| EGA-38 | Add Garage as an optional kind/Helm validation dependency | EGA-37T |
| EGA-38T | Validate the Garage kind/Helm dependency option | EGA-38 |
| EGA-39 | Document Garage operations, licensing, migration, and final decision | EGA-38T |
| EGA-39T | Validate Garage go/no-go artifacts before closing EGA-35 | EGA-39 |

Recommended execution order:

1. Run `EGA-36` and `EGA-36T` first. More provider contract coverage is the
   highest-value risk reducer.
2. Run `EGA-37` and `EGA-37T` next so Garage compatibility is repeatable in
   GitHub Actions.
3. Run `EGA-38` and `EGA-38T` only after the local and CI contract lanes are
   stable.
4. Run `EGA-39` and `EGA-39T` last, because the operations/licensing decision
   should consume the technical compatibility evidence.
